## Getting Started

### What is entity resolution?

Entity resolution (also called record linkage, deduplication, or entity matching) is the process of identifying records across one or more datasets that refer to the same real-world entity, even when the records are not identical. For example, "John Smith" at "123 Main St" and "Jon Smith" at "123 Main Street" likely refer to the same person despite differences in spelling and formatting.

bq-entity-resolution solves this by:

1. **Staging** source records into a common schema
2. **Engineering features** (cleaning names, standardizing addresses, generating phonetic codes)
3. **Blocking** to reduce the comparison space (avoiding all-pairs comparison)
4. **Matching** candidate pairs using configurable comparison functions and scoring
5. **Clustering** matched pairs into groups that represent single entities
6. **Outputting** resolved entities with canonical records

All SQL is generated by Python and executed in BigQuery (or DuckDB for local testing). No data ever leaves your warehouse.

### When should I use this library vs. Splink, dedupe, or other tools?

Use **bq-entity-resolution** when:

- Your data lives in **BigQuery** and you want to keep it there (no data extraction).
- You need a **config-driven** approach where all matching logic is defined in YAML, not code.
- You need **incremental processing** for large-scale production workloads (millions to billions of records).
- You want **progressive disclosure**: start with 5 lines of Python (`quick_config()`), then graduate to full YAML control.
- You need **deterministic SQL generation** that can be previewed, audited, and version-controlled before execution.

Consider **Splink** when you want a Python-first API with Spark or DuckDB backends and strong Fellegi-Sunter support. Consider **dedupe** when you want an active-learning-first approach with small-to-medium datasets in Python memory.

Key differentiators of bq-entity-resolution:

| Feature | bq-entity-resolution | Splink | dedupe |
|---------|---------------------|--------|--------|
| Primary backend | BigQuery | Spark / DuckDB / Athena | Python (in-memory) |
| Config format | YAML (config-driven) | Python API | Python API |
| SQL preview | Yes (`preview-sql`) | Partial | No |
| Incremental processing | Yes (composite watermarks, drain mode) | Limited | No |
| Scale ceiling | 15B+ records (tested) | Depends on backend | ~1M records |
| Checkpoint/resume | Yes | No | No |
| Local testing | DuckDB backend | DuckDB backend | Native |

### Can I use this without BigQuery?

Yes. Install with the local extras:

```bash
pip install "bq-entity-resolution[local]"
```

This installs the DuckDB backend, which requires no Google Cloud credentials and no BigQuery project. You can run the full pipeline locally:

```python
from bq_entity_resolution import Pipeline, quick_config
from bq_entity_resolution.backends.duckdb import DuckDBBackend

config = quick_config(
    bq_project="local-test",
    source_table="local-test.raw.customers",
    columns=["first_name", "last_name", "email"],
)

pipeline = Pipeline(config)
result = pipeline.run(backend=DuckDBBackend())
```

The DuckDB backend supports the same SQL semantics as BigQuery (with automatic SQL adaptation) and is suitable for development, testing, and small datasets. For production workloads above ~100K records, use BigQuery.

## Configuration

### How do I start with a quick config and customize later?

The recommended workflow has three levels:

**Level 1: Auto-config (5 lines)**

```python
from bq_entity_resolution import quick_config

config = quick_config(
    bq_project="my-project",
    source_table="my-project.raw.customers",
    columns=["first_name", "last_name", "email", "phone"],
)
```

`quick_config()` auto-detects column roles, generates features, blocking keys, comparisons, and matching tiers.

**Level 2: Export to YAML for tuning**

```python
yaml_str = config.to_yaml()
with open("my_config.yml", "w") as f:
    f.write(yaml_str)
```

Now you can edit the YAML to adjust weights, add tiers, change thresholds, or add blocking keys.

**Level 3: Load and run the customized config**

```python
from bq_entity_resolution import load_config, Pipeline

config = load_config("my_config.yml")
pipeline = Pipeline(config)
result = pipeline.run(backend=backend)
```

You can also use domain-specific presets as a starting point:

```python
from bq_entity_resolution.config.presets import (
    person_dedup_preset,
    insurance_dedup_preset,
    business_dedup_preset,
)

config = person_dedup_preset(
    bq_project="my-project",
    source_table="my-project.raw.people",
    columns=["first_name", "last_name", "email", "phone", "address_line_1", "zip_code"],
)
```

### What are "roles" and how does auto-detection work?

A **role** is a semantic label assigned to a column that tells the system what kind of data it contains. When you assign a role, the system automatically generates the appropriate features, blocking keys, and comparisons for that column type.

Available roles include: `first_name`, `last_name`, `full_name`, `email`, `phone`, `address_line_1`, `city`, `state`, `zip_code`, `date_of_birth`, `ssn`, `company_name`, `ein`, `policy_number`, `claim_number`, `npi`, `mrn`, `vin`, and more.

**Auto-detection** matches column names to roles using pattern matching:

```python
from bq_entity_resolution.config.roles import detect_role

detect_role("first_name")      # -> "first_name"
detect_role("fname")           # -> "first_name"
detect_role("given_name")      # -> "first_name"
detect_role("email_address")   # -> "email"
detect_role("policy_number")   # -> "policy_number"
detect_role("random_column")   # -> None (no match)
```

When a role is detected or assigned, the system generates:

- **Features**: For `first_name`, it generates `first_name_clean` (via `name_clean`), `first_name_soundex` (via `soundex`), and `first_name_metaphone`.
- **Blocking keys**: For `first_name`, it generates `bk_first_soundex` (via `soundex`).
- **Comparisons**: For `first_name`, it generates a Jaro-Winkler comparison (weight 2.0) and an exact comparison (weight 3.0).

You can override any auto-generated configuration by editing the YAML output.

### How do I add a new column, feature, or comparison?

**Adding a column** -- add it to the `columns` list in your source configuration:

```yaml
sources:
  - name: customers
    columns:
      - name: middle_name        # New column
        type: STRING
        role: first_name         # Optional: assigns a semantic role
```

**Adding a feature** -- add it to `feature_engineering`:

```yaml
feature_engineering:
  name_features:
    features:
      - name: middle_initial
        function: first_letter     # Built-in function
        input: middle_name

      - name: custom_key
        sql: "UPPER(TRIM(my_col))"   # Raw SQL (bypasses registry)
        depends_on: [my_col]
```

**Adding a comparison** -- add it to a matching tier's `comparisons` list:

```yaml
matching_tiers:
  - name: fuzzy
    comparisons:
      - left: middle_initial       # Feature column name
        right: middle_initial
        method: exact              # Built-in comparison method
        weight: 1.5
```

**Adding a custom feature function** (in Python):

```python
from bq_entity_resolution.features import register

@register("my_custom_clean")
def my_custom_clean(inputs: list[str], **_) -> str:
    col = inputs[0]
    return f"REGEXP_REPLACE(UPPER(TRIM({col})), r'[^A-Z0-9]', '')"
```

Then reference it in YAML as `function: my_custom_clean`.

### What matching tiers should I use?

Tiers execute in order. Each subsequent tier only considers pairs not already matched by earlier tiers. A common pattern is:

1. **Exact tier** -- High-precision, low-recall. Match on exact composite keys (e.g., name + address + zip). High threshold, fast execution.

2. **Fuzzy tier** -- Medium-precision, medium-recall. Match on fuzzy string comparisons (Jaro-Winkler, Levenshtein) with multiple blocking paths. Lower threshold.

3. **Probabilistic tier** (optional) -- Uses Fellegi-Sunter log-likelihood scoring with multi-level comparison outcomes. Best for complex matching where simple thresholds are insufficient.

Example:

```yaml
matching_tiers:
  # Tier 1: Exact composite key match
  - name: exact
    blocking:
      paths:
        - keys: [ck_name_address_zip]
    comparisons:
      - left: ck_name_address_zip
        right: ck_name_address_zip
        method: exact
        weight: 10.0
    threshold:
      min_score: 10.0

  # Tier 2: Fuzzy name + address match
  - name: fuzzy
    blocking:
      paths:
        - keys: [bk_last_soundex, bk_zip3]
        - keys: [bk_email_domain]
    comparisons:
      - left: last_name_clean
        right: last_name_clean
        method: jaro_winkler
        weight: 3.0
      - left: first_name_clean
        right: first_name_clean
        method: jaro_winkler
        weight: 2.0
      - left: email_clean
        right: email_clean
        method: exact
        weight: 5.0
    threshold:
      min_score: 7.0
```

Start with one or two tiers and add more only after reviewing match quality. Use `bq-er preview-sql` to inspect the generated SQL before running.

### How do I tune match thresholds?

The threshold controls the minimum score a candidate pair must reach to be classified as a match.

**For sum-based scoring** (`threshold.method: sum`):

Each comparison contributes its `weight` to the total score when the comparison evaluates to true. The threshold is the minimum sum of weights.

To tune:

1. List all comparisons and their weights.
2. Identify the minimum combination of agreements that should constitute a match.
3. Sum those weights to get your threshold.

Example: If `email_exact` has weight 5.0 and `name_jaro_winkler` has weight 3.0, a threshold of 5.0 means email-only matches qualify, while a threshold of 8.0 requires both email and name to agree.

**For Fellegi-Sunter scoring** (`threshold.method: fellegi_sunter`):

The score is a log-likelihood ratio. Use `match_threshold` instead of `min_score`:

```yaml
threshold:
  method: fellegi_sunter
  match_threshold: 6.0    # Log-likelihood ratio threshold
```

Higher values are more conservative (fewer but higher-quality matches). Start around 5.0-8.0 and adjust based on precision/recall review.

**Practical tuning workflow:**

1. Start with a permissive threshold (low value).
2. Run the pipeline and inspect matched pairs in `er_silver.all_matched_pairs`.
3. Look for false positives (incorrect matches) and raise the threshold.
4. Look for false negatives (missed matches) and lower the threshold or add blocking paths.
5. Use active learning (`active_learning.enabled: true`) to surface uncertain pairs for review.

## Performance and Scale

### How many records can this handle?

The pipeline has been tested with datasets exceeding 15 billion records using incremental processing. Guidelines by scale:

| Records | Recommended Approach |
|---------|---------------------|
| < 100K | Default settings. DuckDB backend for testing. |
| 100K - 1M | Add `candidate_limit: 200` to blocking paths. |
| 1M - 10M | Enable table clustering. Set `batch_size`. |
| 10M - 100M | Partition staging tables. Add partition cursors. Tune blocking keys. |
| 100M - 1B | Composite watermarks. INT64 blocking keys. Reserved BigQuery slots. |
| 1B+ | Multi-dimensional watermarks. Drain mode. Partition pruning. Cost ceiling. |

For large-scale production, add scale configuration:

```yaml
scale:
  max_bytes_billed: 10000000000       # 10 GB safety cap per query
  staging_partition_by: "DATE(source_updated_at)"
  staging_clustering: [entity_uid, source_name]
  candidates_clustering: [l_entity_uid]
  matches_clustering: [l_entity_uid, r_entity_uid]
  canonical_index_clustering: [entity_uid]

incremental:
  enabled: true
  cursor_columns: [updated_at, record_id]
  cursor_mode: ordered
  batch_size: 5_000_000
  drain_mode: true
```

### How do I enable incremental processing?

Add an `incremental` section to your YAML config:

```yaml
incremental:
  enabled: true
  cursor_columns: [updated_at]       # Column(s) to track progress
  batch_size: 5_000_000              # Records per batch
  grace_period_hours: 48             # Re-scan window for late-arriving data
```

The pipeline will:

1. Read the watermark (last processed position) from `pipeline_watermarks`.
2. Stage only records beyond the watermark (`WHERE updated_at > watermark`).
3. Process the batch through all pipeline stages.
4. Match new records against historical entities via the **canonical index** (cross-batch blocking).
5. Advance the watermark to the maximum cursor value in the staged batch.

For high-volume sources where many records share the same timestamp, use a **composite ordered cursor**:

```yaml
incremental:
  enabled: true
  cursor_columns: [updated_at, policy_id]  # Composite: timestamp + secondary key
  cursor_mode: ordered                      # Tuple comparison for clean batch boundaries
  batch_size: 5_000_000
```

This generates a tuple comparison `(updated_at, policy_id) > (wm_ts, wm_id)` that guarantees no records are skipped or re-processed.

If no natural secondary column exists, use a hash cursor:

```yaml
incremental:
  hash_cursor:
    column: record_id     # Column to hash
    modulus: 1000          # Number of buckets (0-999)
```

Run `bq-er profile-cursors --config config.yml` to analyze your source data and get cursor strategy recommendations.

### What is drain mode?

Drain mode causes the pipeline to loop through batches until all unprocessed records are consumed.

Without drain mode, a single invocation processes one batch and stops. With drain mode, the pipeline:

1. Processes one batch.
2. Advances the watermark.
3. Checks if more records exist beyond the new watermark.
4. If yes, processes the next batch. If no, stops.
5. Repeats until caught up or `drain_max_iterations` is reached.

Enable it via CLI:

```bash
bq-er run --config config.yml --drain
```

Or in YAML:

```yaml
incremental:
  drain_mode: true
  drain_max_iterations: 100    # Safety cap to prevent infinite loops
```

Drain mode is useful for:

- **Initial backlog processing** after deploying the pipeline.
- **Catch-up after outages** when data accumulated faster than processing.
- **Burst processing** when source data arrives in large batches.

Each iteration is a full pipeline execution with checkpoint support. If any iteration fails, the watermark remains at the last successful position, and the next run resumes from there.

### How do I reduce candidate pair explosion?

Candidate pair explosion occurs when blocking keys are too coarse, generating millions or billions of candidate pairs. Symptoms include slow execution, high BigQuery cost, and potential timeout errors.

**1. Add more selective blocking keys:**

Use composite keys (multiple columns joined together) instead of single-column keys:

```yaml
blocking_keys:
  - name: bk_name_zip
    function: farm_fingerprint_concat
    inputs: [last_name_soundex, zip5]    # Two columns = more selective
```

**2. Lower the candidate limit per blocking path:**

```yaml
blocking:
  paths:
    - keys: [bk_name]
      candidate_limit: 100    # Cap candidates per entity per path
```

**3. Use INT64 blocking keys** (FARM_FINGERPRINT-based) for faster joins:

```yaml
blocking_keys:
  - name: bk_name_zip
    function: farm_fingerprint_concat   # Produces INT64
    inputs: [last_name_clean, zip5]
```

INT64 equality joins are 3-5x faster than STRING joins in BigQuery.

**4. Add multiple blocking paths instead of one coarse path:**

```yaml
blocking:
  paths:
    - keys: [bk_name_zip]       # Path 1: name + zip
      candidate_limit: 200
    - keys: [bk_email_domain]   # Path 2: email domain
      candidate_limit: 200
```

Multiple paths are UNIONed (deduplicated), so a pair only needs to match on one path.

**5. Use hard negatives to disqualify impossible pairs:**

```yaml
hard_negatives:
  - left: entity_type
    right: entity_type
    method: different
    action: disqualify    # Remove pairs where entity types differ
```

## ML and Advanced Features

### How does Fellegi-Sunter probabilistic scoring work?

Fellegi-Sunter is a probabilistic framework where each comparison produces a log-likelihood ratio score instead of a simple pass/fail. It uses two probabilities per comparison level:

- **m probability**: The probability of observing this comparison outcome given the pair is a true match.
- **u probability**: The probability of observing this comparison outcome given the pair is a non-match.

The log-weight for each level is `log2(m / u)`. The total score is the sum of all per-comparison log-weights plus a prior odds term. The match confidence (posterior probability) is computed as `2^score / (1 + 2^score)`.

To enable it on a tier:

```yaml
matching_tiers:
  - name: probabilistic
    threshold:
      method: fellegi_sunter
      match_threshold: 6.0       # Log-likelihood ratio threshold

    comparisons:
      - left: first_name_clean
        right: first_name_clean
        method: jaro_winkler
        levels:
          - label: exact
            method: exact
            m: 0.95             # 95% of true matches have exact first names
            u: 0.01             # 1% of non-matches have exact first names
          - label: fuzzy
            method: jaro_winkler
            params: { threshold: 0.85 }
            m: 0.70
            u: 0.05
          - label: else
            m: 0.05
            u: 0.94
```

**Parameter estimation methods** (instead of manually specifying m/u):

- **EM (unsupervised)**: Learns m/u from candidate pair statistics using Expectation-Maximization, executed entirely within BigQuery scripting.
- **Labeled (supervised)**: Learns m/u from a table of human-labeled pairs.

```yaml
training:
  method: em                    # or "labeled"
  em_max_iterations: 10
  em_convergence_threshold: 0.001
  em_sample_size: 100000
```

### How do I train a BQML model on labeled data?

There are two approaches to using labeled data:

**Approach 1: Fellegi-Sunter with labeled training**

Provide a table of labeled pairs and the system estimates m/u probabilities from them:

```yaml
training:
  method: labeled
  labeled_pairs_table: my-project.labels.match_labels
  parameters_table: my-project.er_silver.fs_parameters    # Persist learned params
```

The labels table schema is: `l_entity_uid STRING, r_entity_uid STRING, is_match BOOL`.

**Approach 2: Active learning feedback loop**

Enable active learning to surface uncertain pairs for human review:

```yaml
matching_tiers:
  - name: probabilistic
    active_learning:
      enabled: true
      queue_size: 500                 # Pairs to review
      uncertainty_window: 0.15        # How close to the boundary
```

The system generates a review queue table with the most uncertain pairs. After labeling, feed them back via the labeled training method.

### How do I use embedding-based matching?

Embedding-based matching uses BigQuery ML to generate text embeddings for records, then uses Locality-Sensitive Hashing (LSH) to find similar records without exhaustive comparison.

```yaml
embeddings:
  enabled: true
  model: text-embedding-004              # BQML embedding model
  source_columns: [first_name_clean, last_name_clean, addr_std]
  concat_separator: " | "
  dimensions: 768
  batch_size: 5000
  lsh:
    num_hash_tables: 20                  # More tables = higher recall
    num_hash_functions_per_table: 8      # More functions = higher precision
    bucket_column_prefix: lsh_bucket
    projection_seed: 42
```

The pipeline will:

1. Concatenate the specified columns with the separator.
2. Generate embeddings using the BQML model.
3. Apply LSH to create bucket assignments.
4. Use LSH buckets as blocking keys for candidate generation.

You can then use `cosine_similarity` or `cosine_similarity_score` as comparison methods in matching tiers:

```yaml
comparisons:
  - left: embedding
    right: embedding
    method: cosine_similarity
    params: { min_similarity: 0.85 }
    weight: 4.0
```

### What clustering algorithms are available?

Three clustering methods are supported:

**1. Connected components** (default, recommended):

Iteratively propagates the minimum cluster ID through match edges until convergence. All entities connected by any chain of matches end up in the same cluster.

```yaml
reconciliation:
  clustering:
    method: connected_components
    max_iterations: 20       # Safety bound for convergence
```

**2. Star clustering:**

Each entity joins the cluster of its highest-confidence match partner. Produces more conservative (smaller) clusters than connected components.

```yaml
reconciliation:
  clustering:
    method: star
```

**3. Best match:**

Each entity is assigned to the cluster of its single best match. The most conservative option -- produces the smallest clusters.

```yaml
reconciliation:
  clustering:
    method: best_match
```

For incremental processing, connected components is recommended because it naturally extends across batches via the canonical index.

## Troubleshooting

### Pipeline validation failed -- what do I do?

Validation errors surface at config load time (before any SQL runs). Common issues:

**"Environment variable '${BQ_PROJECT}' is not set"**

Set the required environment variable:

```bash
export BQ_PROJECT=your-gcp-project
```

Or use a default in your YAML:

```yaml
bq_project: "${BQ_PROJECT:-my-default-project}"
```

**"blocking key 'bk_email' undefined"**

The blocking key referenced in a matching tier is not defined in `feature_engineering.blocking_keys`. Add it:

```yaml
feature_engineering:
  blocking_keys:
    - name: bk_email
      function: farm_fingerprint
      inputs: [email_clean]
```

**"comparison column 'name_clean' not found in features"**

The column referenced in a comparison does not exist as a source column or feature output. Ensure a feature produces it:

```yaml
feature_engineering:
  name_features:
    features:
      - name: name_clean          # This name must match the comparison reference
        function: name_clean
        input: full_name
```

**General debugging workflow:**

1. Run `bq-er validate --config my_config.yml` to get detailed error messages.
2. The validator uses difflib to suggest typo corrections (e.g., "Did you mean 'bk_email'?").
3. Run `bq-er describe --config my_config.yml` to see a human-readable summary of the pipeline configuration.
4. Run `bq-er preview-sql --config my_config.yml --tier exact --stage all` to inspect the generated SQL.

### DuckDB: function not available

**"Catalog Error: Scalar Function SOUNDEX does not exist"** (or similar)

This happens when you use a raw DuckDB connection instead of the `DuckDBBackend` class. The backend registers BigQuery-compatible shims (COUNTIF, TIMESTAMP_DIFF, SOUNDEX, etc.) that are not available in plain DuckDB.

Always use:

```python
from bq_entity_resolution.backends.duckdb import DuckDBBackend

backend = DuckDBBackend()
result = pipeline.run(backend=backend)
```

Do **not** use `duckdb.connect()` directly.

Other DuckDB-specific notes:

- `TO_JSON_STRING(STRUCT(...))` is automatically rewritten to `CAST(struct_pack(...) AS VARCHAR)`.
- `TIMESTAMP_DIFF` arguments are automatically reordered from BigQuery syntax to DuckDB syntax.
- Connected components clustering (which uses BigQuery scripting with `DECLARE`/`WHILE`) is interpreted via a Python loop in the DuckDB backend.
- Jaro-Winkler is implemented via a Python UDF fallback in DuckDB (not a JS UDF).

### Blocking key too coarse / too fine

**Too coarse** (generating too many candidate pairs):

Symptoms: Slow execution, high BigQuery cost, candidate tables with millions of rows.

Solutions:

- Combine multiple columns into composite blocking keys:
  ```yaml
  blocking_keys:
    - name: bk_name_zip
      function: farm_fingerprint_concat
      inputs: [last_name_soundex, zip5]
  ```
- Add `candidate_limit` to blocking paths:
  ```yaml
  paths:
    - keys: [bk_name]
      candidate_limit: 200
  ```
- Use more distinctive blocking functions (e.g., `name_fingerprint` instead of `soundex`).

**Too fine** (missing true matches because blocking keys are too selective):

Symptoms: Low recall, many true matches not found, small clusters.

Solutions:

- Add multiple blocking paths (records only need to match on one path):
  ```yaml
  paths:
    - keys: [bk_last_soundex]      # Path 1
    - keys: [bk_email_domain]      # Path 2 (alternative route)
    - keys: [bk_phone_last4]       # Path 3 (another route)
  ```
- Use coarser blocking functions (e.g., `zip3` instead of `zip5`, `soundex` instead of `exact`).
- Enable cross-batch blocking for incremental pipelines:
  ```yaml
  blocking:
    cross_batch: true
  ```

**Profiling blocking keys:**

Run `bq-er profile --config config.yml` to analyze the distribution of your blocking keys and identify keys that are too coarse (very few distinct values) or too fine (almost unique per record).

### How do I add a custom comparison function?

Register a new comparison function with the `@register` decorator:

```python
from bq_entity_resolution.matching import register

@register("my_custom_comparison")
def my_custom_comparison(
    left: str,
    right: str,
    params: dict | None = None,
    **_,
) -> str:
    """Return a SQL expression that evaluates to TRUE/FALSE or a numeric score."""
    threshold = (params or {}).get("threshold", 0.8)
    return f"my_udf({left}, {right}) >= {threshold}"
```

Then reference it in your YAML config:

```yaml
comparisons:
  - left: name_clean
    right: name_clean
    method: my_custom_comparison
    params: { threshold: 0.9 }
    weight: 3.0
```

The function receives:

- `left`: The left column expression (e.g., `l.name_clean`).
- `right`: The right column expression (e.g., `r.name_clean`).
- `params`: Optional dictionary of method-specific parameters from the YAML config.
- `**_`: Accept extra keyword arguments for forward compatibility.

It must return a valid SQL expression string. For boolean comparisons, return an expression that evaluates to `TRUE` or `FALSE`. For score-based comparisons, return an expression that evaluates to a numeric value.

Built-in comparison functions are documented in the Configuration Reference (`docs/CONFIGURATION.md`). There are 26+ available methods including `exact`, `levenshtein`, `jaro_winkler`, `soundex_match`, `cosine_similarity`, `token_set_match`, `date_within_days`, `numeric_within`, and more.

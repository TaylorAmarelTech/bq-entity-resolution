# Backends Package

## Purpose

Pluggable execution backends implementing the `Backend` protocol. All SQL generated by builders is executed through a backend, enabling BigQuery production use and DuckDB local testing with the same pipeline code.

## Key Files

| File | Description |
|------|-------------|
| `protocol.py` | **Backend protocol** — defines `execute()`, `execute_script()`, `execute_and_fetch()`, `create_table()`, `QueryResult`. All backends implement this interface. |
| `bigquery.py` | **BigQueryBackend** — production backend wrapping `BigQueryClient`. Context manager support (`close()`/`__enter__`/`__exit__`). |
| `bqemulator.py` | **BQEmulatorBackend** — backend for BigQuery emulator (testing). |

## Sub-Package: `duckdb/`

Local development backend with SQL adaptation layer:

| File | Description |
|------|-------------|
| `backend.py` | **DuckDBBackend** — implements Backend protocol using in-memory DuckDB. |
| `sql_adapter.py` | **SQL rewriting** — regex-based transformations from BigQuery SQL dialect to DuckDB: `FARM_FINGERPRINT` → `hash()`, `TIMESTAMP('...')` → `'...'::TIMESTAMP`, `SAFE_DIVIDE` → `CASE WHEN ... THEN ... END`, etc. |
| `scripting.py` | **BQ scripting interpreter** — Python loop that interprets `DECLARE`, `WHILE`, `SET`, `IF`, `BEGIN/COMMIT TRANSACTION` blocks that DuckDB doesn't support natively. |
| `shims.py` | DuckDB function shims: `SOUNDEX`, `REGEXP_CONTAINS`, `FORMAT_TIMESTAMP`, etc. |

## Architecture

```
PipelineExecutor._execute_stage()
  → _is_script_block(sql)?
    ├── Yes → backend.execute_script(sql)
    └── No  → backend.execute(sql)

BigQueryBackend
  → BigQueryClient (clients/bigquery.py)
    → google.cloud.bigquery (with retry, cost tracking, thread-safe)

DuckDBBackend
  → sql_adapter.adapt(sql)    ← regex rewriting
  → duckdb.execute(adapted)   ← or scripting interpreter for WHILE/DECLARE
```

## Key Patterns

- **Protocol-based** — any object implementing `Backend` protocol works. No inheritance required.
- **SQL adaptation** — DuckDB backend rewrites BigQuery-specific SQL via regex patterns. Not 100% coverage but handles the common cases.
- **BQ scripting interpretation** — `WHILE` loops and `DECLARE/SET` variables are parsed and executed in a Python loop since DuckDB doesn't support BQ scripting.
- **Context manager** — BigQueryBackend tracks `_owns_client` to avoid closing externally-provided clients.
- **Thread-safe cost tracking** — BigQueryClient uses `_cost_lock` for concurrent byte accounting.

## Dependencies

- `clients/bigquery.py` — BigQueryClient with retries, cost tracking
- `sql/` — SQLExpression for consistent rendering
- `exceptions.py` — PipelineAbortError for cost ceiling
